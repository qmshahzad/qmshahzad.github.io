Multi-Platform Publishing Automation with n8n

One workflow, one trigger, ten-plus destinations. A single piece of content goes out to WordPress, Substack, Patreon, Medium, Dev.to, Blogger, Wix, GitHub Pages, and Buffer — built by finding a working path into each platform, documented or not.

10+ integrations n8n, Node.js, REST/GraphQL Single build session Error-isolated by design

The problem

A content creator writing one article wants it live on every platform their audience uses — a real blog, a newsletter, a membership page, a dev community, and a social queue — without manually re-pasting it ten times. The catch: most of these platforms either have no public API at all, or an API that only does part of the job (drafts, but not publishing; reading, but not writing).

The job wasn't "call an API." It was figuring out, platform by platform, whether an API existed, what it could actually do, and what to build when it couldn't.

What actually had to be solved

  • Authentication wasn't one thing. Seven different auth models across the project: OAuth2 with refresh tokens, app-level API keys, session cookies, CSRF token pairing, and GitHub's app-installation model. Each platform needed its own credential strategy in n8n, not a copy-pasted pattern.
  • Undocumented protocols had to be read from raw traffic. Medium's content format isn't JSON-with-named-fields — it's a sequence of insert/move/update operations tracking a revision counter. There's no public spec; the only way to learn it was to make a real edit, watch the network request it produced, and infer the rules from the diff.
  • The same error message meant different things on different platforms. A "missing post owner" error on Wix needed a member ID attached to the request. A near-identical authentication failure on a different platform meant the session had simply expired. Diagnosing each one correctly meant not assuming the fix that worked last time would work again.
  • One static site had no backend to call at all. The GitHub Pages portfolio blog wasn't missing an API key — it was missing a CMS, full stop. The fix was building one: a single JSON file as the data layer, read by the existing page's JavaScript, written to by n8n committing files directly through GitHub's API. No database, no hosting cost, fully version-controlled.
  • A failure on one platform couldn't take down the rest. With this many destinations, one platform timing out or rejecting a request needed to be a logged, recoverable event — not a reason the other nine never got their post. The workflow isolates failures per platform and reports a clean summary at the end.

Most of the actual work in automation isn't calling a documented endpoint — it's figuring out what to do when the documentation doesn't exist, the API does half the job, or there's no API to begin with.

What this is good evidence of

Real client automation work rarely arrives as a clean, documented integration. It arrives as "make this work with the tool we already use" — and the tool's API is often incomplete, undocumented, or nonexistent. This project is a working example of diagnosing that gap accurately on ten separate platforms and building the right kind of fix for each one, rather than forcing one technique to fit everything.
